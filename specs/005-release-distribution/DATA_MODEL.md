## Data Model


**Feature**: `005-release-distribution`\
**Phase**: 1 ŌĆö Design\
**Status**: Complete

### Overview

This feature introduces three data artifacts that must be maintained:

1. **Version Record** ŌĆö single source of truth in `deno.json`
2. **Release Manifest** ŌĆö version + platform asset URLs generated per-release
3. **npm Package Graph** ŌĆö 6-node dependency graph of versioned npm packages

There is no persistent database. All state is either in source files (versioned)
or in GitHub Releases (append-only).

---

### Entity 1: Version Record

**Location**: `deno.json` (existing file, extended)

| Field                 | Type   | Required | Example                        | Notes                          |
| --------------------- | ------ | -------- | ------------------------------ | ------------------------------ |
| `name`                | string | Ō£ģ       | `"@myty/claudio"`              | JSR scoped package name        |
| `version`             | string | Ō£ģ       | `"0.2.0"`                      | Semver; single source of truth |
| `exports`             | object | Ō£ģ       | `{ ".": "./src/cli/main.ts" }` | JSR entry point                |
| `compile.permissions` | object | Ō£ģ       | see below                      | Baked into all binaries        |

**Validation rules**:

- `version` MUST be valid semver (`MAJOR.MINOR.PATCH` with optional pre-release)
- `name` MUST match the JSR package name `@myty/claudio`
- `exports["."]` MUST point to the CLI entry point

**Derived constant** (`src/version.ts`):

```typescript
// AUTO-GENERATED by scripts/sync-version.ts ŌĆö do not edit manually
export const VERSION = "0.2.0";
```

This file is regenerated by `scripts/sync-version.ts` on every release and
committed as part of the release commit.

### Entity 3: npm Package Manifest

Each of the 6 npm packages has a `package.json`. Fields marked Ō£ģ are
programmatically written by `scripts/sync-version.ts`.

#### Main Package (`npm/claudio/package.json`)

| Field                  | Type   | Value                      | Written by      |
| ---------------------- | ------ | -------------------------- | --------------- |
| `name`                 | string | `"claudio"`                | static          |
| `version`              | string | `"X.Y.Z"` Ō£ģ               | sync-version.ts |
| `description`          | string | static                     | static          |
| `bin.claudio`          | string | `"./bin/claudio.js"`       | static          |
| `optionalDependencies` | object | all 5 platform packages Ō£ģ | sync-version.ts |
| `engines.node`         | string | `">=18"`                   | static          |
| `os`                   | array  | omitted (any)              | static          |

#### Platform Package (`npm/@claudio/<os>-<arch>/package.json`)

| Field             | Type    | Value                           | Written by      |
| ----------------- | ------- | ------------------------------- | --------------- |
| `name`            | string  | e.g., `"@claudio/darwin-arm64"` | static per-pkg  |
| `version`         | string  | `"X.Y.Z"` Ō£ģ                    | sync-version.ts |
| `os`              | array   | e.g., `["darwin"]`              | static per-pkg  |
| `cpu`             | array   | e.g., `["arm64"]`               | static per-pkg  |
| `preferUnplugged` | boolean | `true`                          | static          |
| `bin`             | object  | `{}` (empty)                    | static          |

**Validation rules**:

- All 6 `version` fields MUST match `deno.json` `version` before publish
- `optionalDependencies` versions MUST match platform package versions
- `preferUnplugged: true` MUST be set on platform packages (required for binary
  resolution)

### Entity 4: Release Asset Manifest

Generated as `release-manifest.json` and attached as a GitHub Release asset
alongside the binaries. Used by tooling that needs to programmatically discover
asset URLs.

```typescript
interface ReleaseManifest {
  version: string; // "0.2.0"
  tagName: string; // "v0.2.0"
  publishedAt: string; // ISO 8601
  assets: ReleaseAsset[];
}

interface ReleaseAsset {
  platform: string; // "macos-arm64"
  filename: string; // "claudio-macos-arm64"
  downloadUrl: string; // "https://github.com/myty/claudio/releases/download/v0.2.0/claudio-macos-arm64"
  sha256: string; // hex-encoded SHA-256 of the binary
}
```

**Example**:

```json
{
  "version": "0.2.0",
  "tagName": "v0.2.0",
  "publishedAt": "2026-03-08T00:00:00Z",
  "assets": [
    {
      "platform": "macos-arm64",
      "filename": "claudio-macos-arm64",
      "downloadUrl": "https://github.com/myty/claudio/releases/download/v0.2.0/claudio-macos-arm64",
      "sha256": "abc123..."
    }
  ]
}
```

This manifest is generated in the `release` job after all binaries are produced,
before uploading to GitHub Releases.

### State Transitions

```
deno.json version bump
  ŌåÆ scripts/sync-version.ts rewrites:
      src/version.ts (VERSION constant)
      npm/*/package.json (version fields)
  ŌåÆ git commit: "chore: release v0.2.0"
  ŌåÆ git tag: "v0.2.0"
  ŌåÆ git push --follow-tags
  ŌåÆ GitHub Actions: release.yml triggered
      ŌåÆ 5 binaries compiled
      ŌåÆ release-manifest.json generated
      ŌåÆ GitHub Release created
      ŌåÆ npm publish (6 packages)
      ŌåÆ deno publish (JSR)
```

### Invariants

1. `deno.json` version = `src/version.ts` VERSION = all npm `package.json`
   version = JSR package version
2. GitHub Release tag = `v{deno.json version}`
3. `claudio --version` output = `"Claudio v{VERSION}"` where VERSION comes from
   `src/version.ts`
4. mise installs the binary whose filename matches the GitHub Release tag
